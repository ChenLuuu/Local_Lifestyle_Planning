# 前端 UI/UX 强制指导规范 · 智能规划阶段专版

> **⚠️ 致 Agent 的最高优先级警告：**
> 绝对禁止在“智能规划阶段”输出类似聊天记录或 Markdown 文本列表的干瘪 UI！
> 必须严格采用**「全屏地图底座 + 悬浮富媒体卡片」**的交互范式。

---

## 核心视觉与交互范式 (Map-Centric Paradigm)

规划生成后，用户第一眼看到的必须是一张**地图**，以及地图上用序号标记的**路线点**。文字信息必须收纳进富媒体卡片中，通过交互（点击/滑动）来展现。

### 1. 视图层级结构 (Z-Index Hierarchy)

组件必须按照以下 Z 轴层级严格堆叠：

- **Z=0 | MapBase (底层)**: 全局背景。在未接入真实地图 API 时，使用一张带有街区纹理的沉浸式伪地图底图（SVG 或 Image），沾满整个视口，带有淡雅的底色（参考“圆周旅迹”风格，如 Tailwind 的 `bg-slate-50` 或 `bg-stone-100`）。
- **Z=10 | RoutePath (连线层)**: 使用 SVG `<path>` 绘制带有流动动效的虚线，按顺序连接各个地点。
- **Z=20 | Markers (节点层)**: 绝对定位在地图上的 POI 标记。使用醒目的圆形图标，内含数字序号（1, 2, 3...）。
- **Z=30 | CardOverlay (交互层)**: 用户点击 Marker 后弹出的「图文详情概览卡片」或触发替换时的「滑卡视图」。

## 关键组件编码规范

### Component 1: `MapMarker` (序号标记点)
- **视觉**：必须是醒目的圆形，带轻微阴影。如 `w-8 h-8 rounded-full bg-yellow-400 text-white font-bold shadow-md`。
- **交互**：点击时触发 Framer Motion 的 `scale` 放大动效，并唤出对应的 `POIDetailCard`。

### Component 2: `POIDetailCard` (地点概览弹窗)
> 解决“枯燥无图文”的核心组件！

- **位置**：可以在对应的 Marker 上方/下方绝对定位弹出，或固定在屏幕下半部分悬浮。
- **布局 (Flex/Grid)**：
  - **顶部/左侧**：必须有高质量的封面图 (`img` 标签，使用 `object-cover`，圆角 `rounded-t-xl` 或 `rounded-xl`)。
  - **内容区**：
    - **主标题**：地点名称（加粗，大字号）。
    - **副标题/Tags**：核心卖点（如“适合拍照”、“低卡”、“距您 200m”），使用 `bg-gray-100 text-gray-600 px-2 py-1 rounded-md text-xs`。
    - **简介**：1-2 句核心描述。
  - **操作区**：必须包含一个醒目的**「换一个 (Replace)」**按钮，点击后进入滑卡模式。

### Component 3: `SwipeReplacementCard` (替补滑卡视图)
- **触发条件**：用户在 `POIDetailCard` 中点击“换一个”或对该节点表示不满意时全屏/半屏覆盖居中弹出。
- **动效 (Framer Motion)**：
  - 绑定 `drag="x"`。
  - 左滑（x < 阈值）：卡片飞出，显示红色负反馈，加载下一个替补方案。
  - 右滑（x > 阈值）：卡片飞出，显示绿色正反馈，确认替换，更新底层 Map 的节点数据。
- **内容要求**：与 `POIDetailCard` 类似，必须是**强图文**结构，绝不能是纯文本。

## 验收标准 (DoD)

Agent 在提交本阶段代码时，必须对照检查：
1. [ ] 屏幕主体是否为地图（哪怕是假图）？
2. [ ] 是否彻底移除了纯文本的纵向时间线列表？
3. [ ] 每一个 POI 节点是否都包含了图片展示槽位（`<img>` 或 `div.bg-image`）？
4. [ ] 点击节点是否能弹出包含图文的详细卡片？
