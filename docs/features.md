## P0 核心功能（MVP · 黑客松必须完成）

---

### F01: 渐进式需求收集 UI — 步骤1+2（单选 + 词云多选）

- **行为**：用户打开页面，首先看到单选卡片序列（玩伴→人数→地点→场景→预算），每次单选后自动进入下一题；完成单选后，根据选择结果动态生成气泡词云（如选「闺蜜」→ 生成「出片/美食/艺术/美甲/购物/不想走路/室内」等标签），用户可多选；前端将单选结果 + 多选标签合并为结构化 JSON 传给后端
- **验证命令**：`pytest tests/unit/test_ui_collect_step1_2.py -v`
- **状态**：`passing`
- **依赖**：无
- **证据**：`23 passed in 0.31s`（2026-05-28）

---

### F02: 渐进式需求收集 UI — 步骤3（特殊需求文本框 + 提交）

- **行为**：在词云多选完成后，展示自由文本输入框，用户可补充特殊需求（如「宠物友好/不吃辣/学生党/薅羊毛」）；点击「开始规划」后，将三步信息合并为完整 ConstraintSet（硬约束：max_distance_km / age_range / total_duration；软偏好：noise_level / per_capita / tags[]）提交给规划 Agent
- **验证命令**：`pytest tests/unit/test_ui_collect_step3.py -v`
- **状态**：`passing`
- **依赖**：F01
- **证据**：`46 passed in 0.27s`（2026-05-28）

---

### F03: 核心规划 Agent（ReAct 主循环）

- **行为**：`POST /api/plan/run` 接收完整 ConstraintSet，启动 ReAct 循环（Thought → Action → Observation），asyncio 并行调用 venue_search / restaurant_search / route_plan，串行调用 check_availability，最终返回包含 3-4 个节点的行程 JSON（含时间轴 / 地点 / 预计花费 / 交通方式）；SSE 流式输出每步推理过程；方案生成时长 ≤ 30 秒
- **验证命令**：`pytest tests/integration/test_react_loop.py -v`
- **状态**：`passing`
- **依赖**：F02
- **证据**：`12 passed in 3.70s`（2026-05-28）

---

### F04: 动态时间分配算法

- **行为**：给定活动列表和用户时间窗口，执行三步分配：① 硬性校验（计算最小耗时+交通矩阵，超窗口按优先级裁剪）→ ② 弹性分配（剩余 slack 时间按权重分配给可延伸项）→ ③ 冲突检测（拓扑验证无重叠无间隙，失败则触发重排）；输出严格不超时、无重叠的时间线
- **验证命令**：`pytest tests/unit/test_time_allocator.py -v`
- **状态**：`passing`
- **依赖**：F02（需要 ConstraintSet 中的 total_duration 字段）
- **证据**：`26 passed in 0.12s`（2026-05-28）

---

### F05: 节点级滑卡微调（Tinder Swipe 机制）

- **行为**：初始方案生成后，用户可对任意单一节点（如「第二站的餐厅」）点击「不满意」；Agent 在保持该节点前后行程时间/地理位置不冲突的前提下，生成 3 个替补活动卡片；用户左滑（跳过）/ 右滑（接受），直到接受一个或放弃；接受后整体方案时间线自动重算，确认无冲突；支持对同一节点反复微调直到满意
- **验证命令**：`pytest tests/integration/test_node_swap.py -v`
- **状态**：`passing`
- **依赖**：F03, F04（需要已有行程方案 + 时间重算能力）
- **证据**：`17 passed in 0.36s`（2026-05-28）

---

### F06: 批量事务执行（一键确认并执行）

- **行为**：所有节点经用户逐一确认后，页面出现「确认并一键执行」按钮；`POST /api/execute` 触发后，asyncio 并发调用全部 Mock API（模拟订座 / 买票 / 生成打车路线 / 外卖下单）；幂等键防止重复下单；逐条返回执行状态（进度条实时更新）；端到端流程 ≤ 2 分钟；全部成功后输出确认文案
- **验证命令**：`pytest tests/e2e/test_execute_booking.py -v`
- **状态**：`passing`
- **依赖**：F05（所有节点滑卡确认完成后才触发执行）
- **证据**：`19 passed in 0.29s`（2026-05-28）

## P1 重要功能（强烈推荐完成）

---

### F07: 异常处理与兜底（Risk Agent · 三类故障三级降级）

- **行为**：Mock 层以 10% 概率注入三类故障（无座/无票/时间冲突）；fault_handlers 在规划生成和执行两个阶段均可触发，统一路由：Level 1 静默替换（同地点换时段 / 同区域换场馆，对用户透明展示替换原因）→ Level 2 方案重排（拓扑排序重算顺序 / 压缩 buffer）→ Level 3 暂停执行通知用户选择（周边 500 米内无同类 / 超出决策边界时）；任何故障不向用户抛出代码级崩溃
- **验证命令**：`pytest tests/unit/test_fault_handlers.py -v`
- **状态**：`passing`
- **依赖**：F03（规划阶段需要）、F06（执行阶段需要）—— 两个阶段均需覆盖
- **证据**：`26 passed in 0.16s`（2026-05-28）
